## なぜ今、円ステーブルコインが増えているのか

日本では2023年6月、改正資金決済法の施行によって、法定通貨と連動し額面で償還できるステーブルコインが「電子決済手段」として制度上に位置づけられた。ここが出発点だ。

その後、資金移動業者が発行する JPYC や、信託銀行が発行する JPYSC など、異なる法的スキームを採用する円建てステーブルコインが具体化していく。さらに2026年6月の改正では、信託型ステーブルコインの裏付け資産の運用範囲の拡大や、電子決済手段・暗号資産サービスの仲介業といった制度が整備され、発行・流通の実務環境が一段進んだ。

一見すると、これらはすべて「1トークン＝1円」の似た商品に見える。だが、発行主体・利用者資金の保全方法・発行や償還の額の制約・外部ウォレットへの移し方は、採用する法的スキームによって変わる。Zennには既に[4種を整理した記事](https://zenn.dev/komlock_lab/articles/japan-yen-stablecoin-landscape-2026)があるので、本記事はそれと役割を分け、法スキームの違いが発行・償還フローやオンチェーン設計にどう効くかに絞って掘る。

:::message
本記事は制度と実装の整理であって、法的助言ではない。実際の発行や設計にあたっては、金融規制に詳しい専門家の確認を通してほしい。
:::

## 前提：どのスキームに絞って話すか

まず言葉を1つだけ。「電子決済手段」とは、改正資金決済法が定めた、いわば「デジタルの円」の正式名称だ。円を裏付けに発行され、誰に対しても支払いに使え、誰とでも売買できるトークンを指す。「1コイン＝1円で使える、法律にちゃんと位置づけられた円のトークン」と思えばいい。

発行の作り方は、発行主体で見ると大きく3つある。

| 発行主体 | 型 | ざっくり |
| --- | --- | --- |
| 銀行 | 預金型 | 銀行預金をベースにしたトークン |
| 資金移動業者 | 未達債務型 | 送金業者が自分の債務として発行 |
| 信託会社・信託銀行 | 信託型（3号電子決済手段） | 信託銀行が発行し、企画者は委託者に回る |

本記事では、国内の事例で具体化が進んでいる「資金移動業者発行型」と「信託型」の2つに絞って比べる。制度上は銀行預金を基礎とする形態もあるため、厳密には完全な二択ではない点は先に断っておく。あわせて、資金決済法上の「1号／3号」という条文上の分類と、ビジネス上の「資金移動業型／信託型」は完全な同義ではない（銀行預金ベースのトークンも要件次第で1号に入り得る）。ここでは実務での呼び方に寄せて話を進める。

## 資金移動業者発行型 vs 信託型

この2つは「誰が法律上の発行者になるか」で性質が変わる。まずイメージをつかもう。

- 資金移動業者発行型：自分（発行会社）が資金移動業の登録を取り、自分が発行者になる。送金業者の免許を自前で持つ格好だ。
- 信託型（3号）：信託銀行に発行してもらい、自分は「企画者」に回る。利用者が払った円は信託銀行が信託財産として預かり、会社のお金とは分けて管理する。この信託財産が受託者の固有財産から法的に分離される点が特徴で、これを倒産隔離と呼ぶ。

主な違いを並べるとこうなる。ただし後述するように、「移転の自由度」は発行スキームだけで一意に決まるわけではない。

| 観点 | 資金移動業者発行型 | 信託型（3号） |
| --- | --- | --- |
| 法的な発行者 | 発行会社自身 | 信託銀行・特定信託会社 |
| 企画者のライセンス | 資金移動業の登録が必要 | 不要（企画者・委託者に回る） |
| 利用者資金の保全 | 供託・保証・信託等の法定保全措置 | 信託財産として受託者の固有財産から分離 |
| 取り扱える取引額 | 第二種は1件100万円相当以下／第一種は上限なし | 原則上限なし（特定信託会社は認可で100万円超） |
| 外部ウォレットへの移転 | 商品・取扱業者の設計による | 商品・取扱業者の設計による |
| パーミッションレス流通 | 制度上ただちに否定されない | 設計しやすいとされるが自動的に可能になるわけではない |

保全について補足する。資金移動業者発行型は「無保全で発行会社の一般債権者になる」わけではない。供託・保証・信託などの法定の保全措置が課され、2026年改正で新しい保全方法も整備された。一方の信託型は、信託法上の倒産隔離が働く。どちらも利用者資金は守られる設計で、守り方の仕組みが違う、と捉えるのが正確だ。

裏付け資産の運用も2026年改正で広がった。信託型の裏付けは従来ほぼ要求払預金に限られていたが、一定の短期国債や中途解約できる定期預貯金について、所定の上限と元本毀損を防ぐ要件のもとで運用が認められた（[BUSINESS LAWYERS](https://www.businesslawyers.jp/articles/1476)）。準備金が利回りを生む余地ができた形だが、その利回りがそのまま発行体の収益になるかは信託契約や受益権の構造による。

## 大前提：発行スキームと流通スキームは別物

ここが誤解の温床なので、独立して置いておく。「どのスキームで発行したか」と「オンチェーンでどう流通するか」は、直結しない。

発行スキームが決めるのは、発行者は誰か・利用者資金をどう保全するか・発行や償還にどんな額の制約がかかるか、といった点だ。一方で、外部ウォレットへ出庫できるか、どのチェーンで流すか、移転を許可制にするかは、AML/CFT方針、電子決済手段等取引業者の管理方法、償還受付の設計、対応ウォレット、チェーン上のリスク管理によって決まる。つまり商品設計の話だ。

「資金移動業型だから必ず許可制」「信託型だから必ず自由移転」という対応関係は、成り立たない。

## エンジニア目線：スキームは設計にどう効くか

法制度を踏まえて、実装で気にすべき点を3つに絞る。

### 移転制御は「法律の一律要件」ではなく「商品設計」

まず誤解を解く。資金移動業者発行型だからといって、全transferで受取アドレスがKYC済みかを検証するホワイトリストが法律上必須になるわけではない。むしろ電子決済手段は不特定多数の間で移転・売買できる性質が求められるため、KYC済み利用者間だけに閉じたトークンは電子決済手段に該当しない可能性がある、という整理がある。Progmatも資金移動業型について「KYC未済アドレスへの移転可」と整理している（[Progmat](https://progmat.co.jp/20231128_blog/)）。

もちろん、発行・販売・交換・償還・カストディを担う事業者には、本人確認、マネロン対策、トラベルルール、制裁対象アドレスへの対応が必要だ。ただしそれを、ERC-20の全移転にallowlistを埋め込む義務、と読み替えるのは誤りになる。

そのうえで、発行体が商品設計としてあえて許可制移転を採る場合はある。その実装は、たとえば移転時に本人確認済みかを検証する形になる。

```solidity
// 【許可制移転を商品設計として採用した場合の例】
// 法律上の一律要件ではなく、あくまで発行体の設計判断
// OpenZeppelin v4系のフックを想定（v5では _update が相当）
function _beforeTokenTransfer(address from, address to, uint256 amount) internal override {
    // ミント（from == 0）とバーン（to == 0）以外に移転制約を課す場合
    if (from != address(0) && to != address(0)) {
        require(kycRegistry.isVerified(to), "recipient not allowed");
    }
    super._beforeTokenTransfer(from, to, amount);
}
```

要は「許可制にするなら移転そのものにゲートを埋め込む」という話で、許可制にするかどうかは法律ではなく設計で決まる。信託型は倒産隔離や外部流通を設計しやすいとされるが、これも自動的にパーミッションレスになるわけではない。

### 100万円の制約が「かかる場所」

送金上限は、オンチェーンの1回のtransfer額に一律でかかるものではない。第二種資金移動業者が取り扱える為替取引が、原則1件100万円相当以下に限られる、という業務区分の話だ。つまり制約がかかるのは、発行体からの購入や発行体への償還など、資金移動業者が直接引き受ける取引の設計であって、利用者がセルフホストウォレットに出庫した後のオンチェーン移転すべてに、スマートコントラクト上の100万円チェックが必須になるわけではない（[Progmat](https://progmat.co.jp/)）。

ここを区別しないと、「ERC-20のtransferに100万円上限を入れなければならない」という誤った実装に走る。100万円超の取引を正面から扱いたいなら、第一種資金移動業や、信託型＋認可といった道が選択肢になる。

### ミント・バーンと円償還の突合

基本フローは、円入金でミント、償還でバーンだ。ここは両スキーム共通で、裏付け資産の管理主体が違う（資金移動業者は供託等で保全、信託型は信託銀行が信託財産として分別管理）。実装の勘所は、オンチェーンのバーンとオフチェーンの円払い戻しをどう突き合わせ、償還の真正性をどう保証するかにある。

## ケースで見る：JPYC・JPYSC・XJPY

実在／構想を、スキームと提供状況で並べるとこうなる。

| コイン | スキーム | 提供状況 | 補足 |
| --- | --- | --- | --- |
| JPYC | 資金移動業者発行型 | 提供中 | 資金移動業者登録を取得、1 JPYC＝1円で償還可能。信託型JPYCも共同検討 |
| JPYSC | 信託型（3号） | 提供中（限定） | SBI新生信託銀行が発行、SBI VCトレードで提供。公開時点で外部入出庫は未対応 |
| XJPY／XUSD | 信託型（Progmat基盤） | 構想・共同検討 | 三菱UFJ信託／Progmat。取引所間決済の効率化を想定 |

JPYCは資金移動業者発行型で1円償還を実現しつつ、三菱UFJ信託・Progmatと組んで信託型JPYCも共同検討している（[JPYC](https://corporate.jpyc.co.jp/news/posts/mufg_progmat)）。同じブランドでもスキームを使い分けようとしている。

JPYSCは信託型の3号電子決済手段で、2026年6月24日にSBI VCトレードで提供が始まったが、公開時点では外部ウォレットへの入出庫に対応しておらず、口座内での売買に限定されている。技術開発はStartaleが担い、独自のレイヤー1構想も示されている。「信託型だからただちにパーミッションレスで流通する」わけではない好例だ。JPYSCとそれを発行するSBIの狙いは別記事で扱った。

https://zenn.dev/yuta1995/articles/sbi-crypto-vertical-integration-bitbank

XJPY／XUSDは、確認できる範囲では構想・共同検討の段階として説明されている。Progmat Coin自体は、Ethereumなどの上で動くマルチチェーン対応の発行・管理基盤だ（[Progmat](https://progmat.co.jp/20231128_blog/)）。発行体ごとに別チェーンを立てるのではなく、共通基盤の上に載せる設計になっている。

## どちらを選ぶか：見るべき5つの軸

「小口なら資金移動業型、高額なら信託型」と単純化すると外す。発行スキームと流通スキームが別物である以上、少なくとも次の5軸で見る必要がある。

1. 発行・償還の額（100万円の制約がどこにかかるか、第一種か第二種か）
2. 利用者資金の保全構造（供託・保証・信託か、信託財産の分離か）
3. 発行主体のライセンス（自社で取るか、信託銀行に発行してもらうか）
4. 外部ウォレットと流通範囲（口座内限定か、パブリックチェーンで流すか）
5. KYC・AML・カストディを誰が担うか（発行体・販売者・取引業者・カストディが別法人になることもある）

信託型でも電子決済手段等取引業者との連携や、信託受益権の設計・信託報酬・受託審査が要る。資金移動業者発行型でも外部ウォレットやパブリックチェーン対応は可能で、第一種なら100万円超も扱える。だからこそ、スキーム名だけで決めず、この5軸で具体的に詰めるのが実務だ。

## まとめ

- 法定通貨連動ステーブルコインを「電子決済手段」と定めたのは2023年6月施行の改正資金決済法。2026年6月改正は、信託型の裏付け資産運用の拡大や仲介業の整備が中心
- 発行主体は銀行・資金移動業者・信託会社の3つ。本記事はうち具体化が進む資金移動業者発行型と信託型に絞った（厳密には完全な二択ではない）
- 発行スキームと流通スキームは別物。移転が許可制かパーミッションレスかは、法律ではなくAML方針や商品設計で決まる
- 100万円の制約はオンチェーンの全transferではなく、資金移動業者が引き受ける取引にかかる。ERC-20に一律の上限チェックを入れる話ではない
- 利用者資金はどちらのスキームでも保全される。守り方（供託・保証・信託か、信託財産の分離か）が違う

次は、この規制の土台そのものが動く話に触れたい。暗号資産の規制を資金決済法から金融商品取引法へ移す改正が進んでおり、制度の前提が変わっていく。ここは稿を改める。
